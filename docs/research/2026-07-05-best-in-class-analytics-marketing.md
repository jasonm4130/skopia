### 2.6 README patterns ranked by consistency (n=5 exemplars + counterscale)

1. **Centered logo + sub-10-word category-defining tagline in the first viewport (5/5).** Format: "The [adjective] open-source [category] platform" or "[Name] is a [adjectives] alternative to [incumbent]" — naming the incumbent does the positioning in one line.
2. **Badge row immediately under the tagline (5/5), badge *content* is a strategy choice.** License is the only universal badge; the rest split into liveness (stars, commits/month, Docker pulls, CI), social proof (Twitter, HN points), or CTA (Umami's green demo badge).
3. **Features before any install content (4/5 — all but Umami), always as *linked claims*,** never bare bullets (Plausible: prose blocks → marketing pages; Supabase: checkboxes → docs).
4. **Hero product screenshot in the first 1.5 screens (4/5 — all but Umami),** after pitch/features, before any code block. For a dashboard product the screenshot is the argument.
5. **The commercial/hosting model stated openly (5/5 where one exists;** Cal.diy states the *absence*: "There is no hosted/managed version."). Plausible and Dub turn the explanation itself into a trust asset.
6. **Tech stack as a named list (4/5)** — cheap credibility with contributors.
7. **Install pushed out of the README or far down (4/5)** — the stronger the cloud business, the thinner the in-README install. Self-host-first products (Umami, Counterscale, Cal.diy) are the exception: install is the point, and it's high up.
8. **Contributor-activity proof near the bottom (3/5):** avatar walls, Repobeats. **None of the six uses a star-history chart** — that's an earlier-stage momentum pattern (per [star-history.com's playbook](https://www.star-history.com/blog/playbook-for-more-github-stars/); see also [Infisical's 3K-stars post](https://infisical.com/blog/how-we-got-3000-github-stars-in-2-months), [Lago's first-1000-stars post](https://www.getlago.com/blog/how-we-got-our-first-1000-github-stars) — all credit screenshot-above-fold, incumbent-naming one-liner, live demo, launch-day HN badge rather than README length).
9. **Distribution loops embedded in the README (2/5 but high-leverage):** Supabase's watch-releases GIF + "Made with" badges; Umami's demo badge; Dub's frozen HN badge.

**Analytics-specific:** both analytics exemplars put the **live demo as the first interactive element** (Umami: badge row; Plausible: first item of its link row), and for Umami the demo fully substitutes for a screenshot. Public-share-dashboard-of-your-own-site is the standard demo mechanism (Plausible, Simple Analytics, Umami all dogfood it; Counterscale runs a dedicated demo subdomain).

## 3. Launch moments — HN threads

All comment trees mined via the Algolia items API (`hn.algolia.com/api/v1/items/{id}`), not search snippets.

### 3.1 Counterscale — no Show HN ever happened

Algolia full-text, URL-restricted, and author searches all confirm Counterscale's Feb 2024 GitHub launch **never produced a Show HN**. The only substantive thread is Ben Vinegar's retrospective blog post, submitted by Armin Ronacher four months later: **"Counterscale and the New Self-Hosted"** — 2024-06-24, 29 points, 31 comments — [news.ycombinator.com/item?id=40777196](https://news.ycombinator.com/item?id=40777196). Lesson zero for Skopia: the closest comp forfeited its launch moment entirely; a 2K-star ceiling followed.

- **Praised:** the abstraction level ("there is no other way to get things like analytics engine, queues… for the price they offer along with $0 for bandwidth egress" — vmg12).
- **Attacked:** (a) **the "self-hosted" label** — candiddevmike: "My definition of self-hosted isn't the same as your definition"; jorams: "Everything about it is built on non-portable, proprietary products exclusively available from and hosted by Cloudflare." (b) **Cloudflare trust** — electrodank: "I'm not comfortable hosting anything on Cloudflare"; localfirst citing the $120k-shakedown story. (c) **free-tier flexing dismissed** — mrits: "100k requests being free is an odd flex."
- **Founder replies:** conceding worked — "Maybe self-operated is better terminology — that's fair" — and honest tradeoff framing landed ("It comes with serious constraints (like a dependency on Cloudflare). But some people may prefer those tradeoffs"). His ClickHouse-needs-8GB cost strawman backfired ("I have kafka and clickhouse running on a 2GB vm" — mrits).

### 3.2 Plausible — three threads, one masterclass

**(a) "An alternative to using Google Analytics on your website"** — 2020-04-08, 462 points, ~301 comments — [item?id=22813168](https://news.ycombinator.com/item?id=22813168). Praised for listing competitors honestly and putting themselves last ("This type of 'honorable' advertising is OK in my book" — y-c-o-m-b). Attacked for: stealth-ad framing ("a sleazy ad in disguise" — vikramkr); factual overreach on GA's data use (lmkg, a GA consultant: "The article's use of that verbiage is deceptive… Google is a Processor under GDPR"); pricing-vs-free-GA; and "open source" without a self-host path ("So basically you ask people to work for free on something only you can use?" — shubidubi). markosaric fixed the pricing nav *mid-thread*; the self-host complaint was answered six months later by shipping it:

**(b) "Show HN: Plausible – Self-Hosted Google Analytics alternative"** — 2020-10-06, 351 points, 139 comments — [item?id=24696145](https://news.ycombinator.com/item?id=24696145). Praised: the 0.7 KB script, MIT tracker, "a gulp of fresh air." Attacked:
- **Daily salt rotation kills retention/conversion data** — "The daily salt reset is a deal breaker… every conversion will seem it took less than 24 hours" (bamazizi); "'how many unique visitors do you get in a month'… you're saying your solution cannot answer this" (propogandist). *(Skopia uses the same daily-salt technique — this objection arrives verbatim.)*
- **IP+UA hashing accuracy** — "a workplace with 5000 identically imaged desktops… will incorrectly under-report unique users" (gnabgib).
- **GDPR transmission gotcha** — "Since users of Plausible are sending you their visitors IP addresses, doesn't that mean GDPR applies?" (malisper).
- **Adblocker irony** — plausible.io itself was blocked for some commenters ("They're listed in your pihole… had to override the block" — corobo).
- **"Not a GA alternative"** — no outbound links/conversions "was a deal breaker" (nicbou).

**Founder replies (the playbook):** markosaric pasted one canonical mechanism answer repeatedly — `hash(daily_salt + website_domain + ip_address + user_agent)` + "old salts are deleted" + [plausible.io/data-policy](https://plausible.io/data-policy) — owned limitations flatly ("no. that's the decision we made in order to be as privacy focused as possible"), positioned instead of apologizing ("the 20% of GA features that 80% of GA users need"), and converted criticism into artifacts live ("i've added the request to our github now").

**(c) "Plausible Analytics Isn't GDPR Compliant"** — 2020-10-23, 54 points — [item?id=24868012](https://news.ycombinator.com/item?id=24868012). A hostile third-party fingerprinting accusation. ukutaht cited GDPR Recital 26 and linked the actual salt-rotation source; neutral experts then defended Plausible unprompted ("what Plausible doing is good and compliant" — donohoe). **Consistent, linkable documentation meant the community litigated on their behalf.**

### 3.3 Umami — the concession case study

The actual Show HN (2020-08-17) flopped at 10 points ([item?id=24184773](https://news.ycombinator.com/item?id=24184773)); a repost the next day exploded: **"Umami: Self-hosted open-source alternative to Google Analytics"** — 2020-08-18, **820 points, 227 comments** — [item?id=24198329](https://news.ycombinator.com/item?id=24198329).

- **Praised:** speed, polish, the 30-day solo build; self-hosting beating adblockers framed as a feature.
- **Attacked:** the **demo 502'd under HN load** (brutal for an analytics tool); **GDPR-claim demolition** — malisper audited the source ("it claims it doesn't use cookies, but it clearly uses localStorage to store a 'sessionKey'"); "If you don't feel fit to judge whether something breaches GDPR, then maybe you shouldn't say 'so it is GDPR and CCPA compliant'" (madeofpalk); "not really a GA alternative"; dependency-count nitpicks.
- **Founder replies:** mcao's first defense ("I don't think any analytics product can exist" under strict GDPR) drew more fire; then he folded completely and **edited the website mid-thread** ("I went ahead and removed any mention of GDPR from the website") — and the hostile subthread instantly flipped ("Thank you for removing GDPR mentions, but mostly for building this in the first place!" — Aeolun).

### 3.5 The 5 recurring objections Skopia's launch assets must pre-empt

Ranked by frequency across all mined threads:

1. **"Why not just use X?"** (every thread; usually the first comment). Matomo, GoAccess/server logs, GoatCounter, "no analytics at all" — and for a Cloudflare-native tool, the sharpened variant: **"isn't this just Cloudflare Web Analytics?"** (nreece was already asking this in Plausible's 2020 thread; Plausible and Fathom both maintain vs-cloudflare pages because the question converts). Pre-empt: a "Skopia vs Cloudflare Web Analytics" section (retention, custom events, multi-site, data ownership, UTM breakdowns) + one honest FAQ line each for GoatCounter / Plausible CE / Counterscale.
2. **Privacy/GDPR-claim skepticism** — hashed IP+UA "is still fingerprinting"; "you still need consent"; "If it's unique, it's PII" (y42, [item?id=24198329](https://news.ycombinator.com/item?id=24198329)). Claiming compliance without a documented mechanism is the trap; the documented mechanism (Plausible) turned the identical attack into free advocacy. Pre-empt: a Plausible-grade data-policy page with the exact HMAC formula, salt-rotation schedule, and source links; **never write the words "GDPR compliant"** (the Umami lesson).
3. **"Privacy features gut the data businesses actually need"** — daily salt kills monthly uniques/retention/conversion attribution; "not a GA alternative"; "can't tell new from old customers is a non-starter for most businesses" (chrshawkes). Pre-empt: own it as positioning ("the 20% of GA that 80% need"), and lead with Skopia's honest-metrics UI (the dashboard already caveats its Count column) as the differentiator rather than hiding the limitation.
4. **Self-host definition, friction, and TCO** — "My definition of self-hosted isn't the same as your definition"; "non-portable, proprietary products exclusively available from Cloudflare"; conversely, true self-hosting gets hit with TCO/downtime warnings ("it's extremely easy to underestimate [TCO]" — WhitneyLand). Pre-empt: say **"self-deployed on your own Cloudflare account"** (Ben Vinegar's crowdsourced fix), state the Cloudflare dependency as an explicit tradeoff in the launch post's first screen, and make the four-secret setup friction honest and scripted.
5. **Accuracy attacks from both directions + the adblocker irony + Cloudflare trust** — NAT/VPN under-counting; the privacy tool itself getting adblocked (verify `skopia.js` and the demo domain aren't on EasyPrivacy before launch — plausible.io was blocked *in its own thread*); "I don't trust Cloudflare" as the deploy-platform variant. Pre-empt: publish honest accuracy caveats (already Skopia's brand), first-party-domain serving as the adblock answer, and don't flex Cloudflare's free tier — state real cost math instead.

### 3.6 Three founder-reply tactics that visibly worked

1. **Concede fast and change the artifact mid-thread** (mcao editing umami.is live; markosaric adding a pricing link and filing an issue mid-thread; bentlegen adopting "self-operated"). Defending first always made it worse (pauljarvis's snark defense, mcao's "no analytics product can exist," bentlegen's ClickHouse strawman).
2. **One canonical, linkable mechanism answer** — paste the same hash formula + data-policy URL every time; link the actual source file. Neutral experts then do the defending for you.
3. **Own the limitation as positioning, never dispute the fact.** The framings that never survived HN contact: "GDPR compliant," "self-hosted" on someone else's proprietary cloud, "GA alternative" without conversions. Every overclaim became the thread's top attack; every honestly-framed tradeoff got defended by commenters.

## 5. Skopia gap notes (from local repo review)

Reviewed before making recommendations, so they're gap-aware:

- **"Live demo" CTA points at a login wall.** `skopia-www` Hero and Cta components link "Live demo" / "See the live demo" to `https://app.skopia.dev` — but the product has **no public/share dashboard** (`analytics/src/` has no share-token route; the dashboard is auth-gated behind the owner password, and the feature roadmap item 7 in `docs/specs/2026-07-03-feature-roadmap.md` confirms even `/live` is auth-gated).
