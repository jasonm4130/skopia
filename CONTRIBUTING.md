# Contributing to Skopia

Thanks for your interest in Skopia — privacy-respecting, self-hostable web analytics
that runs entirely on the Cloudflare developer platform.

Before you start, skim [`CLAUDE.md`](./CLAUDE.md): it is the operating contract for
this repo (the design principles, the non-goals, and where decisions are recorded).
Most of what follows is a practical summary of it.

## Prerequisites

- **Node.js** 20+.
- **pnpm** — this repo pins its package manager in `package.json`
  (`packageManager: pnpm@…`). Enable it with `corepack enable`, or install via your
  version manager of choice.
- A Cloudflare account is only needed to *deploy*; you can develop and run the test
  suite without one.

## Getting set up

```sh
pnpm install
cp .dev.vars.example .dev.vars   # fill in the four values for `wrangler dev`
pnpm dev                         # runs the Worker locally; reads .dev.vars
```

Never commit `.dev.vars` — it is gitignored and holds your local secrets.

The repository layout is documented in the [README](./README.md#repository-layout).

## Conventions

- **TypeScript, strict mode**, everywhere.
- **Biome** owns formatting and linting. Before committing:
  ```sh
  pnpm lint:fix   # auto-fix + format
  pnpm lint       # verify clean
  ```
- **Match the surrounding style.** Surgical changes only — every changed line should
  trace to the thing you are fixing or building. Don't reformat or "improve"
  adjacent code in the same PR.
- **Do not hand-edit generated files.** `src/shared/skopia-embed.ts` (the minified
  tracking script) and `src/shared/schema-embed.ts` (the D1 DDL) are regenerated by
  `pnpm build`. Edit the sources (`src/script/skopia.ts`, `migrations/`) instead.

## The tracking-script budget is sacred

The browser script in `src/script/skopia.ts` ships to **every visitor of every
tracked site**, so its size is a product guarantee, not a nice-to-have:

- **Hard budget: < 2 KB gzipped**, enforced by `pnpm check:size` (part of the pre-PR
  `pnpm run ci` gate).
- **No cookies / `localStorage` / `sessionStorage` / `indexedDB`** — audited by
  `pnpm check:cookieless`.
- **No external/runtime dependencies** in the script or Worker bundle
  (`pnpm check:no-external`).

Treat a regression in any of these as a bug. If you think you need to cross the
budget, that's a discussion to have in an issue first.

## Tests

- We use **Vitest** with the Cloudflare Workers pool
  (`@cloudflare/vitest-pool-workers`).
- Write a failing test first for non-trivial logic, then make it pass.
- Run the suite with `pnpm test` (or `pnpm test:watch` while iterating).

## Before you open a PR

Run the full gate and make sure it is green:

```sh
pnpm run ci
```

This runs typecheck + lint + tests + build + the three script audits. (Use
`pnpm run ci`, not `pnpm ci` — the latter is pnpm's clean-install built-in.) Quote
the green result in your PR description.

Guidelines for the PR itself:

- One concern per PR. Keep it focused and reviewable.
- Write a clear description: what changed and why.
- If your change deviates from an approved spec in `docs/specs/`, update the spec or
  open a new ADR in `docs/decisions/` (one decision per file, dated, with context +
  consequences). Architecture and roadmap direction are owned by the project's
  planning agents — large changes should align with a spec or ADR rather than
  arriving unannounced.

## Reporting bugs and security issues

- **Bugs / feature ideas:** open a GitHub issue.
- **Security vulnerabilities:** do **not** open a public issue — follow
  [`SECURITY.md`](./SECURITY.md).
